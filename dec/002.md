# 002 — HTTP Library Choice

## Status: Accepted

## Context
The script needs to make HTTP requests to the GitHub API (metadata queries and zip archive downloads). The HTTP library must work on Windows, support HTTPS, handle bearer token authentication, expose response headers (for rate limit tracking), and stream large responses to disk (zip downloads can be tens of megabytes).

## Decision
**WinHTTP** — the native Windows HTTP library included in the Windows SDK.

## Alternatives Considered

| Library | Rejected Because |
|---------|-----------------|
| libcurl | Cross-platform abstraction layer. Adds an external dependency. WinHTTP calls do not map 1:1 to NASM equivalents because libcurl wraps WinHTTP internally. |
| WinINet | Older Microsoft HTTP library. Lacks the connection pooling, timeout resolution, and header access granularity that WinHTTP provides. Microsoft itself recommends WinHTTP over WinINet for services. |
| Raw sockets (Winsock) | Would require implementing TLS, HTTP/1.1, chunked transfer encoding, and gzip decompression manually. Enormous scope for no benefit — these protocols are solved problems. |

## Rationale
WinHTTP is a native Windows system library — no external dependencies, no installation required. It provides HTTPS support, bearer token authentication via headers, response header access for rate limit tracking, and streaming response bodies to files. In NASM, WinHTTP API calls translate directly to `stdcall` calls with the correct calling convention — no wrapper layer needed.
