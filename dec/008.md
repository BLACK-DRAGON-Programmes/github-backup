# Decision 008 — Immediate Shutdown on 'q' Key (Spec Override)

## Date
2026-06-18 (R153)

## Context
Spec Section 11 originally stated: "the daemon finishes the current repository download (if in progress), writes the cycle summary, closes the log file, releases the mutex, and exits."

Sir (BLACK_DRAGON) tested the software on Windows and found that pressing 'q' in the viewer did NOT kill the daemon immediately. The daemon continued downloading a 90MB repo for 10+ seconds after 'q' was pressed. This caused:
1. The viewer to exit (after its 5-second wait) while the daemon was still running.
2. The backup.exe file to remain locked by the running daemon process.
3. Recompile attempts to fail with "Permission denied" (R152).

Sir's instruction: "It should actually kill the process. It should shut down the entire app in its entirety. It should end everything."

## Decision
**Override spec Section 11.** The daemon now checks for shutdown DURING active downloads (not just between repos). When shutdown is requested mid-download:
1. The download streaming loop (`stream_zip_to_file`) detects the shutdown via `ctx->should_stop()`.
2. The download is aborted immediately (returns -5).
3. The `.zip.tmp` temp file is cleaned up by `backup_single_repo` (atomic_write pattern — old `.zip` backup remains intact).
4. The backup cycle is aborted (`cycle_aborted = 1`).
5. The daemon exits cleanly.

## Safety
This is safe because of the atomic_write pattern (Decision 005):
- Downloads write to `repo.zip.tmp` (temp file).
- Only after successful download + verification does `atomic_write` rename `.tmp` → `.zip`.
- Aborting mid-download just deletes the `.tmp` file. The old `.zip` backup is never touched.
- No corruption risk. No data loss.

## Implementation
- `backup.h`: Added `BACKUP_SHUTDOWN` to `backup_result` enum.
- `network.c:stream_zip_to_file()`: Added `ctx->should_stop()` check at top of each chunk-loop iteration. Returns -5 on shutdown.
- `network.c:try_download_once()` + `download_repo_zip()`: Propagate -5 (existing `return result` at line 1263 handles this).
- `backup.c:backup_single_repo()`: Maps download_repo_zip return -5 → `BACKUP_SHUTDOWN`. Calls `cleanup_temp_file` before returning.
- `backup.c:run_backup_cycle()`: `BACKUP_SHUTDOWN` case sets `cycle_aborted = 1`, breaks the repo loop.

## Rationale
The original spec's "finish current download" behavior was designed to avoid incomplete zip files. However, the atomic_write pattern already protects against this — the temp file is deleted on abort, and the old backup is never touched. The "finish download" behavior caused operational problems (file lock preventing recompile, daemon running headless after viewer closed). Sir's instruction takes precedence per AGENTS.md: "Sir's own suggestions are auto-approved (R9)."
