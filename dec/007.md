# 007 — Two-Process Architecture (Daemon + Viewer)

**Date:** 2026-06-05
**Status:** Accepted

## Context

The tool needs to run as a persistent background process (Task Scheduler, system startup) while also providing on-demand console visibility. The user wants to:

1. Close the console window at any time without stopping the backup process.
2. Open a console at any time (via double-click or toast click) to see live progress.
3. Shut down the backup only through an explicit action (pressing `q` or `--shutdown`).

The previous single-process model could not support this because Windows terminates console processes when their console window is closed — `CTRL_CLOSE_EVENT` always ends the process regardless of handler return value. The `FreeConsole()` approach (R72) detached the console but meant the process could never get a console back, and closing the terminal window still killed the process if it had one.

## Decision

Adopt a **two-process architecture**: a **daemon** process (headless, owns the mutex, runs the backup loop) and **viewer** processes (disposable, tail the log file, can be opened and closed freely).

### How it works

1. **Daemon** (`backup.exe --daemon`): Launched with `CREATE_NO_WINDOW`. Owns the `Global\GitHubBackupMutex`. Runs the full backup loop headless — log file output + toast notifications only. Never has a console.

2. **First double-click**: `backup.exe` (no flags) tries to create the mutex. Succeeds (no daemon) → spawns `backup.exe --daemon` as a detached process, releases mutex, enters viewer mode.

3. **Subsequent double-clicks / toast clicks**: `backup.exe` tries to create the mutex. Fails (daemon owns it) → enters viewer mode directly.

4. **Closing a viewer**: Only the viewer process dies. The daemon is unaffected. The user can open a new viewer anytime.

5. **`q` key in viewer**: Opens the shutdown event (`Global\GitHubBackupShutdown`), signals the daemon. Daemon exits gracefully. Viewer exits after daemon.

6. **`--shutdown`**: Opens and signals the shutdown event. Same mechanism as `q`, usable from scripts.

### Key rules

- The daemon NEVER has a console. It is always headless.
- The daemon owns the mutex. Viewer processes never own it.
- Closing a viewer NEVER kills the daemon.
- `Ctrl+C` in a viewer closes only the viewer (not the daemon).
- Only `q` (in viewer) or `--shutdown` (command line) stops the daemon gracefully.

## Consequences

- The `--background` flag becomes obsolete — replaced by `--daemon` (always headless).
- `Ctrl+C` no longer triggers daemon shutdown — it only closes the viewer.
- The PowerShell toast script must register an `Activated` event handler to launch `backup.exe` on click.
- The spawn-on-first-click logic adds a `CreateProcess` call to the viewer startup path.
- Multiple viewers can coexist — each independently tails the log file.
- `taskkill /IM backup.exe /F` force-kills everything (daemon + viewers) — non-graceful, documented as last resort.

## Alternatives considered

1. **Single process + `FreeConsole()`/`AllocConsole()`**: Cannot reattach a console after `FreeConsole()` without `AllocConsole()`, which creates a blank console with no scrollback. Also, closing the `AllocConsole()` window still terminates the process.

2. **Windows Service**: Overkill for a simple periodic backup tool. Requires installation, service control manager, elevated permissions for `CreateService`. Cannot easily show a console window for ad-hoc viewing.

3. **Named pipe IPC for console relay**: The daemon could write to a named pipe and a viewer could read from it. More complex than log file tailing and provides no practical benefit — the log file already serves as the communication channel.
