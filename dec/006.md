# 006 — Single-Instance, Log Viewer, Shutdown, and Pretty Console

## Status: Accepted

## Date: 2026-06-04

## Context

The operator requested four related features: the tool should run in the background without a terminal, detect if an instance is already running, provide a live log viewer when a second instance is launched, support a shutdown command to stop the running instance, and display pretty colored console output when a terminal is attached.

## Decision

**Single-executable, three-mode architecture** — all features are implemented within the single `backup.exe`. No separate scripts or helper processes.

**Mode selection at startup:**
1. Parse command-line arguments first.
2. If `--shutdown` flag is present → signal the running instance to exit and terminate.
3. If `--background` flag is present → run backup mode, detach console after init via `FreeConsole()`.
4. Attempt to create a named mutex (`Global\GitHubBackupMutex`).
   - Mutex created → enter **backup mode** (normal operation).
   - Mutex exists → enter **log viewer mode** (tail the log file with ANSI colors).

**Shutdown mechanism:**
- The running backup instance creates a named event (`Global\GitHubBackupShutdown`).
- `backup.exe --shutdown` opens this event and sets it.
- The backup loop checks the event at the start of each cycle iteration.
- If signaled: finish current repo download (if in progress), write cycle summary, close log, release mutex, exit.
- Ctrl+C in foreground mode also triggers graceful shutdown via SIGINT handler.

**Console output:**
- ANSI escape codes for color (VT100 via `SetConsoleMode` + `ENABLE_VIRTUAL_TERMINAL_PROCESSING`).
- Color scheme: INFO=gray, OK/SUCCESS=green, WARN=yellow, ERROR=red, timestamps=dim gray, repos=cyan.
- Column-aligned format with fixed-width fields.
- In background mode (detached console), console output is suppressed.
- Log viewer tails from current EOF — no history, only new entries.

## Alternatives Considered

| Approach | Rejected Because |
|----------|-----------------|
| Separate PowerShell wrapper script | Adds an external dependency. Violates the "single .exe, no runtime dependencies" requirement. The NASM version would need to replicate the wrapper. |
| Named pipe for IPC | Overkill for a simple shutdown signal. Named events are the idiomatic Win32 approach for single-signal IPC. |
| File-based signal (e.g., `backup.stop` file) | Race conditions, file cleanup issues, and requires polling. Named events are atomic and kernel-managed. |
| Rich console UI (ncurses-style) | Disproportionate complexity. The log viewer is a passive tail, not an interactive UI. |
| Background as a Windows service (SCM) | Massive complexity increase. The tool already runs fine via Task Scheduler. Service registration, SCM interaction, and service control codes add NASM translation burden for no practical benefit. |

## Rationale

All four features are interdependent: background running needs shutdown support, shutdown needs instance detection, instance detection enables log viewer, and log viewer needs pretty console output. Implementing them as one coherent unit is simpler than adding them piecemeal. The named mutex + named event pattern is idiomatic Win32 — both objects are kernel-managed, have no resource leaks, and translate directly to NASM (CreateMutex, CreateEvent, WaitForSingleObject).
