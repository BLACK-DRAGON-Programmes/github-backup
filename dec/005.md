# Decision 005 — Atomic Backup Write Behavior

## Date

2026-05-25

## Status

Accepted

## Context

The original specification (Section 5) stated: "If a previous zip for this repo exists, overwrite it." This was ambiguous about the overwrite mechanism — it could be interpreted as opening the file for write (which destroys the old content immediately) or as a safe replacement.

The intended behavior was clarified: the script must NEVER delete the old backup before the new backup is confirmed on disk and accessible. During a single backup cycle, there should never be a state where no valid backup exists for a repository.

## Decision

Backup writes must be atomic using a temp-file-then-rename pattern:

> **Note:** The directory path `D:\BACKUP\` in the steps below is the default value, configurable at runtime via the `BACKUP_DIR` variable in `.env`.

1. Download the repository zip to a **temporary file** (e.g., `D:\BACKUP\<repo-name>.zip.tmp`)
2. Verify the temporary file is fully written, valid, and accessible on disk
3. Delete the previous backup file (`D:\BACKUP\<repo-name>.zip`) — only if it exists
4. Rename the temporary file to the final path (`D:\BACKUP\<repo-name>.zip`)

If the download fails or the file is corrupted, the temporary file is deleted and the previous backup remains intact. The script never enters a state where a repository has no backup.

## Rationale

This is an application of the atomic writes coding standard. The concern is data loss: if the script overwrites the old backup and the download fails midway, both the old and new backups are lost. The atomic approach guarantees that a valid backup always exists on disk.

This also addresses a concern about "multiple copies of the same repository from different timestamps" — the script never creates timestamped copies. Only one backup per repo exists at any time, and the old one is only removed after the new one is confirmed.

## Error Cases

- **Download fails:** Temporary file is deleted. Old backup remains. Toast fires with error details. Continue to next repo.
- **Disk full during write:** Temporary file is deleted. Old backup remains. Toast fires with error. Cycle stops per spec Section 7.
- **Verification fails (corrupt zip):** Temporary file is deleted. Old backup remains. Toast fires with error. Continue to next repo.

## Impact on File Plan

No new files required. The atomic behavior is implemented within the existing `src/backup.c` module. The temp file path and rename logic are internal to the backup write function.

## Spec Changes

- Section 5 (Backup Logic): Rewrite step 5c (save zip) to describe the atomic temp-file-then-rename pattern instead of simple overwrite
- Section 7 (Error Handling): Update disk full behavior to clarify that the temporary file is cleaned up and the old backup remains
