# 001 — Language Choice

## Status: Accepted

## Context
The GitHub Backup Script needs an intermediate language before the final NASM assembly rewrite. The language must produce a single static `.exe` with no runtime dependencies, use Win32 API calls directly, and serve as a mechanical stepping stone to NASM.

## Decision
**C (compiled with MinGW-w64 on Windows).**

## Alternatives Considered

| Language | Rejected Because |
|----------|-----------------|
| PowerShell | High abstraction level. Does not map to NASM Win32 calls. Requires .NET runtime. |
| Go | Produces statically-linked binaries but operates at a high abstraction. Go runtime, goroutines, garbage collector have no NASM equivalent. |
| Rust | Ownership system, trait objects, standard library abstractions — none map to NASM. Would require significant design divergence from the eventual assembly version. |
| Python | Spec explicitly excludes Python (Section 6). Interpreted, requires runtime, no .exe without bundler. |

## Rationale
C's Win32 API calls (WinHTTP for HTTP, Kernel32 for file I/O) map almost 1:1 to NASM x86-64 Win32 calls. A C implementation can be translated to NASM mechanically — replace standard library calls with Win32 API calls and libc functions with hand-written routines. No other language in consideration offers this property.
