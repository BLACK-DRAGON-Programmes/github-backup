# 003 — Log Rotation Strategy

## Status: Accepted

## Context
The script writes log entries to a single file (`backup.log`) appended after each backup cycle. Over time, this file grows unbounded. A rotation strategy is needed to prevent the log from consuming excessive disk space.

## Decision
**Delete and restart** — when the log file exceeds the configured size threshold (`LOG_MAX_SIZE_BYTES`), delete the entire file and start fresh. No archiving, no compression, no timestamped copies.

## Alternatives Considered

| Strategy | Rejected Because |
|----------|-----------------|
| Append with size cap (stop writing) | Silent log silence — the operator would not know the log stopped accepting entries. Violates Coding Standard #4 (errors must be loud). |
| Timestamped rotation (backup.log.1, backup.log.2) | Multiple files to manage. Adds complexity for an ephemeral log. The NASM implementation would need filename generation logic for no practical benefit. |
| Compressed archiving (backup.log.gz) | Requires a compression library. The NASM version would need a deflate implementation. Disproportionate complexity for a disposable log. |

## Rationale
The log is ephemeral — it exists for operational debugging, not as an audit trail. GitHub's API responses are deterministic and repeatable. If something goes wrong, the fix is in the code or the .env, not in historical log data. Deleting and restarting is the simplest strategy that preserves the log's usefulness while requiring zero additional logic in the C or NASM implementation. A single `DeleteFile` call followed by reopening for append.

## Implementation Notes
- Check file size after each cycle complete log entry
- If size > `LOG_MAX_SIZE_BYTES`, call `DeleteFile` then reopen
- If `LOG_MAX_SIZE_BYTES` is 0, rotation is disabled (log grows indefinitely)
- The rotation threshold is configurable via `.env` — operators with high cycle frequency or who want longer history can increase the value
